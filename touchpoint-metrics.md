# 触点指标 · Touchpoint Metrics

> **指标 ID**：`tp-id-xxx` · **触点类别**：静态/契约/样本/工具/版本/反馈六类 · **测评**：Agent / 测试 / 专家走查

列说明：**维度**（易获取性、易学、一致性、准确性、易实践性、易用性、兼容性、完整性等）· **量化指标** · **打分规则** · **测评方法**

---

## 资料/文档

**定义：** README、安装文档、教程、API 手册、架构文档、FAQ、排障指南、贡献文档等所有静态文本知识载体。

**特征：** 单向静态信息媒介；以文字/图表为载体；开发者通过主动阅读、检索、理解完成知识输入；无实时交互，依赖认知解码完成任务。

### 易获取性

#### tp-id-001 · 检索命中率

关键词搜索能否 1 次定位到目标内容，包括搜索轮次、搜索时长等。完成一个标准任务需要翻阅文档跳转次数。

- **Agent 口径：** 文档可发现性——搜索关键词与结果匹配度分析；高频任务关键词命中率 = 预设 N 个关键词（如「内存申请」「流同步」「事件」）在 docs 中可定位到对应章节的比例
- **打分：** ≥95% → 10 分；每降 5% 扣 1 分
- **测评：** Agent

#### tp-id-002 · 文档跳转次数

README 到任意 API 详情页的最少点击/链接跳转数。

- **打分：** ≤2 跳 → 10 分；每增加 1 跳减 1 分
- **测评：** Agent

#### tp-id-003 · 多入口可达率

可同时通过 README / 目录 / 索引三种方式到达的接口比例。

- **打分：** ≥98% → 10 分；每降 5% 扣 1 分
- **测评：** Agent

### 易学 / 易理解（认知负荷）

阅读理解、信息梳理消耗的心智成本；概念/术语与友商相比数量。

#### tp-id-004 · 单次任务文档跳转浏览次数

- **打分：** 单次 → 10 分；每增加 1 次扣 2 分
- **测评：** Agent

#### tp-id-006 · FAQ 命中率

近 N 个 Issue 中可被现有 FAQ 解答的比例。

- **打分：** ≥90% 满分；每降 5% 扣 1 分
- **测评：** Agent

### 一致性

粒度与开发者经验一致性：深浅内容分层，新手/专家各取所需。文档结构一致性：同类文档结构与前后版本风格一致；内容与版本时效一致，没有过时信息等。

#### tp-id-007 · 文档结构风格一致率

quick_start、dev_guide、FAQ、troubleshooting、release_notes 等与模板一致率 / 总文档数。

- **打分：** 100% → 10 分；每降 2% 扣 1 分
- **测评：** Agent

#### tp-id-008 · 概念跨文档冲突数

同一概念在 quick_start / dev_guide / api_ref 中描述出现矛盾的数量。

- **打分：** 0 处 → 10 分；每出现 1 处扣 1 分
- **测评：** Agent

### 准确性

页面内容描述错误、命令错误、版本不匹配占比。

#### tp-id-011 · 内/外链有效率

可正常跳转的链接数 / 总链接数（脚本扫描）。

- **打分：** ≥98% → 10 分；每降 2% 扣 1 分
- **测评：** Agent

#### tp-id-012 · 文档错误点位密度

代码/描述/拼写存在错误文档数 / 总文档数（Agent 扫描）。

- **打分：** ≥98% → 10 分；每降 2% 扣 1 分
- **测评：** Agent

### 易实践性

基于文档，开发者从零到跑通 readme、Demo 是否能成功及时长。

#### tp-id-017 · 一次跑通成功率

跟随 quick_start、HelloWorld、sample 一次运行成功。

- **打分：** 100% 一次跑通 → (8,10] 分；≥90% → (6,8]；70%～89% → (4,6]；50%～69% → (2,4]；＜50% → [0,2]
- **测评：** Agent

## API / 接口

**定义：** 对外暴露函数、结构体、枚举、环境变量、配置项、扩展接口等编码契约型交互入口。

**特征：** 强契约、强约束、可编程交互；是开发者心智映射到代码的直接载体；具备版本依赖、语义约束、调用依赖；属于高心智门槛、长期复用型触点。

### 易用性（易理解、易调用）

对接程序员既有心智编程模型应尽可能复用已有思维模式，减少认知转换成本。成功的编程模型是在现有心智模型基础上增加必要的并行抽象（如 CUDA SIMT 让开发者以「标量思维」编写并行代码）。

#### tp-id-030 · 接口自解释读懂率

API 命名语义直观性：函数/类命名是否直观；参数命名是否自解释；命名是否与数学/算法概念对应。

- **打分：** 100% 可自解释 → (8,10] 分；≥90% → (6,8]；70%～89% → (4,6]；50%～69% → (2,4]；＜50% → [0,2]
- **测评：** Agent

## 源码 & 示例类

**定义：** 核心源码、examples 样例、Demo 工程、模板工程、适配层代码。

**特征：** 可阅读、可运行、可模仿、可复用；从理论到实践的具象载体；开发者通过模仿、复刻、溯源完成学习与问题定位；高实践性、高参考性。

### 易学 / 易用（认知负荷）

代码逻辑清晰、分层合理、关键逻辑有注释；新手无需额外查阅资料即可读懂设计思路；样例低耦合、模板化程度高，可直接拷贝、裁剪、改造迁移。

#### tp-id-043 · 编程模型标准对齐率

样例执行流程是否严格遵循 CANN 标准编程时序（初始化 → 建流 → 申请内存 → 数据拷贝 → 计算 → 同步 → 释放）。

- **打分：** 完整编程模型点位对齐 → 10 分；每丢失 1 个对齐点扣 2 分
- **测评：** Agent

## 工具

**定义：** 编译工具、诊断工具、拓扑探测、Benchmark、容器镜像、包管理、安装脚本、环境预检工具、性能工具、精度分析工具。

**特征：** 自动化、封装化、黑盒交互；替开发者屏蔽底层复杂细节；以命令行/脚本/镜像为交互形态；降低操作负荷、屏蔽环境差异。

### 易获取性

软件分发工程实践：云原生、AI 异构生态（昇腾、CUDA、TensorRT）均采用多仓分发 + 国内加速源 + 离线包组合策略，降低跨区域、隔离环境的获取成本。

#### tp-id-046 · 有效获取渠道数

可正常获取工具的官方仓库、镜像站、包管理器等渠道总数。

- **打分：** ≥5 个 → 10 分；5 个以下每少 1 个扣 2 分
- **测评：** Agent

#### tp-id-047 · 检索平均轮次

查找工具资源的平均搜索次数。信息检索理论：检索轮次直接反映信息可发现性，轮次越多，信息寻址成本越高。

- **打分：** 1 次 → 10 分；每增加 1 次扣 2 分，最低 2 分
- **测评：** Agent

### 开箱即用性（交互负荷）

工具依赖自检完备、环境适配广、操作链路短，新手零上手成本。

#### tp-id-048 · 首次安装/部署一次性成功率

标准环境下，工具安装 / 镜像启动 / 脚本执行首次成功比例。

- **打分：** 100% 成功 → 10 分；每降低 5% 扣 0.5 分，最低 2 分
- **测评：** Agent

#### tp-id-049 · 前置依赖手动补充率

需要开发者自行安装依赖、配置环境变量、修改配置文件的场景占比。

- **打分：** 0% → 10 分；每提升 10% 扣 1 分，最低 2 分
- **测评：** Agent

### 易用性（交互负荷）

预制场景、模板或默认值等，尽量减少用户输入或操作交互难度和成本。

#### tp-id-050 · 平均操作步数

- **打分：** ≤3 步 → 10 分；每增加 1 步扣 1.5 分，最低 2 分
- **测评：** Agent

### 一致性

风格、交互行为一致，使用成本低；跨环境工具行为无突变；同类工具/场景设计理念、使用习惯对齐。

#### tp-id-052 · 标杆/场景交互范式一致度

与标杆工具、或同类型场景操作逻辑、使用流程的一致程度。

- **打分：** 完全重合 → 10 分；每降低 5% 扣 1 分，最低 2 分
- **测评：** Agent

## 版本发布包

**定义：** Release、Tag、分支、Changelog、安全公告、废弃声明、版本兼容矩阵。

**特征：** 时间维度生命周期契约；具备时序性、延续性、风险传导性；开发者用于版本选型、升级决策、风险评估、长期运维；属于决策型触点。

### 易用性

聚焦操作成本、阅读成本、使用效率，反映交互负荷。

#### tp-id-067 · 版本部署命令数

完成版本安装、镜像拉取、代码拉取的独立命令总行数。

- **打分：** ≤3 条 → 10 分；4 条 → 8 分；5 条 → 6 分；6 条 → 4 分；≥7 条 → 2 分
- **测评：** Agent

#### tp-id-068 · 版本切换配置修改量

升降级版本时需手动改动的配置、脚本、环境项数量。

- **打分：** 0 项 → 10 分；1 项 → 8 分；2 项 → 6 分；3 项 → 4 分；≥4 项 → 2 分
- **测评：** Agent

### 兼容性

版本间适配关系明确，升级、降级、跨版本协作稳定，破坏性变更可控，分支与版本间依赖合规，适配多类软硬件环境。

#### tp-id-069 · 版本配套硬件兼容性率

版本兼容不同硬件比例。

- **计算：** 硬件兼容性率 = 实际可正常适配硬件数 ÷ 官方声明支持硬件总数 × 100%
- **打分：** 每降 5% 扣 2 分
- **测评：** Agent

## 运行反馈

**定义：** 运行日志、错误提示、错误码、告警信息、性能输出、编译报错。

**特征：** 系统单向即时反馈媒介；日志、报错、告警为排障核心线索。

### 易获取性

运行日志、报错、错误码、告警、性能数据输出位置明确、调取路径短、无需额外配置/权限；开发者可快速查看、导出、检索反馈信息，无访问壁垒。

#### tp-id-071 · 反馈信息调取平均命令数

查看日志、编译报错、告警、性能输出所需点击/输入指令数。

- **打分：** 命令数 ≤2 → 10 分；每增加 1 条扣 2 分
- **测评：** Agent

### 完整性

单条反馈信息要素齐全，全场景反馈覆盖无缺失；日志、报错、告警附带必要上下文、环境信息、调用链路，支撑完整问题定位与分析。

#### tp-id-075 · 错误码配套信息完整性

错误码配套原因、临时方案、根治办法、FAQ 齐全的占比。

- **打分：** 100% 完备 → 10 分；每降低 5% 扣 2 分，最低 2 分
- **测评：** Agent
